## 1. 개요

에이전트 팀은 여러 Claude Code 인스턴스가 협력해서 작업하는 구조다.  
한 세션이 **팀 리드(lead)** 역할을 맡아 작업을 조율하고, 나머지 **팀원(teammate)** 들은 각자의 컨텍스트 창에서 독립적으로 작업하며 서로 직접 메시지를 주고받는다.

**가장 효과적인 상황:**
- 병렬 탐색이 실질적인 가치를 더할 때
- 리서치/리뷰: 여러 팀원이 동시에 서로 다른 측면을 조사
- 새 모듈/기능 개발: 팀원마다 독립된 파트 소유
- 경쟁 가설 디버깅: 팀원이 서로 다른 이론을 병렬로 검증
- 프론트엔드·백엔드·테스트처럼 계층 간 변경 작업

**적합하지 않은 상황:**
- 순차적 작업 (의존성이 많은 단계별 처리)
- 같은 파일을 여러 에이전트가 수정해야 할 때
- 단순하고 빠른 작업 (토큰 비용 대비 효율이 낮음)

## 2. 서브에이전트 vs 에이전트 팀 비교

|  | 서브에이전트 | 에이전트 팀 |
|---|---|---|
| **컨텍스트** | 자체 컨텍스트 창, 결과만 상위에 반환 | 자체 컨텍스트 창, 완전 독립 |
| **통신** | 메인 에이전트에게만 결과 보고 | 팀원끼리 직접 메시지 가능 |
| **조율** | 메인 에이전트가 모든 작업 관리 | 공유 태스크 리스트로 자율 조율 |
| **적합한 상황** | 결과만 중요한 집중적 단일 작업 | 토론·협업이 필요한 복잡한 작업 |
| **토큰 비용** | 낮음: 결과가 메인 컨텍스트로 요약 | 높음: 팀원마다 별도 Claude 인스턴스 |

> **핵심 차이**: 팀원끼리 직접 소통이 필요하면 에이전트 팀, 결과만 받으면 되면 서브에이전트.

## 3. 활성화 방법

`~/.claude/settings.json`에 추가:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

또는 환경변수로:

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

> Claude Code 재시작 필요. 버전 확인: `claude --version` (v2.1.32 이상)

## 4. 팀 시작하기

자연어로 팀 구성을 요청하면 Claude가 팀을 생성하고 팀원을 스폰한다.

**예시 프롬프트:**

```
CLI 도구를 설계 중입니다. 에이전트 팀을 만들어서 다른 각도로 탐색해주세요:
- 한 팀원은 UX 담당
- 한 팀원은 기술 아키텍처 담당
- 한 팀원은 악마의 변호인(반론 제기) 역할
```

팀 리드 터미널에서 팀원 목록과 각자 작업 중인 내용이 표시된다.  
`Shift+Down`으로 팀원을 순환하며 직접 메시지를 보낼 수 있다.

## 5. 팀 제어

### 5-1. 화면 표시 모드

| 모드 | 설명 | 요구사항 |
|---|---|---|
| **in-process** | 모든 팀원이 메인 터미널 안에서 실행, `Shift+Down`으로 전환 | 없음 (모든 터미널 지원) |
| **split panes** | 팀원마다 별도 창, 동시에 출력 확인 가능 | tmux 또는 iTerm2 필요 |

기본값(`auto`)은 tmux/iTerm2 환경이면 split panes, 아니면 in-process.

`~/.claude/settings.json`에서 고정 설정:
```json
{
  "teammateMode": "in-process"
}
```

단일 세션 강제 지정:
```bash
claude --teammate-mode in-process
```

### 5-2. 팀원 수와 모델 지정

```
4명의 팀원으로 팀을 만들어서 이 모듈들을 병렬로 리팩토링해줘.
각 팀원은 Sonnet 모델을 사용해.
```

팀원은 기본적으로 리드의 모델을 상속하지 않는다. `/config`의 "Default teammate model"에서 설정 가능.

### 5-3. 계획 승인 요구 (Plan Approval)

```
인증 모듈을 리팩토링할 아키텍트 팀원을 스폰해줘.
변경하기 전에 계획 승인을 요구해.
```

- 팀원이 계획 완료 → 리드에게 승인 요청 전송
- 리드가 승인 또는 피드백과 함께 반려
- 반려 시 팀원이 계획 수정 후 재제출
- 승인 후 실제 구현 시작

승인 기준을 프롬프트에 명시 가능: `"테스트 커버리지를 포함한 계획만 승인해"`

### 5-4. 팀원과 직접 대화

- **in-process**: `Shift+Down`으로 팀원 전환 후 메시지 입력, `Ctrl+T`로 태스크 리스트 토글
- **split panes**: 팀원 창 클릭 후 직접 입력

### 5-5. 태스크 할당과 자율 클레임

공유 태스크 리스트로 작업 조율.  
태스크 상태: `pending` → `in progress` → `completed`

- **리드 할당**: 리드가 특정 팀원에게 명시적으로 태스크 배정
- **자율 클레임**: 팀원이 자신의 태스크 완료 후 다음 미할당 태스크를 스스로 가져감
- 파일 락킹으로 동시 클레임 충돌 방지

### 5-6. 팀원 종료

```
researcher 팀원에게 종료 요청해줘
```

팀원이 승인하면 정상 종료, 거부하면 이유와 함께 반려.

### 5-7. 팀 정리 (Cleanup)

```
팀 정리해줘
```

- **반드시 리드가 실행** (팀원이 실행하면 리소스 불일치 발생 위험)
- 활성 팀원이 있으면 정리 실패 → 먼저 팀원 종료 후 정리

### 5-8. 품질 게이트 (Hooks)

| Hook | 트리거 | 코드 2 반환 시 |
|---|---|---|
| `TeammateIdle` | 팀원이 유휴 상태가 되려 할 때 | 피드백 전송 후 팀원 계속 작업 |
| `TaskCreated` | 태스크 생성 시 | 생성 차단 후 피드백 전송 |
| `TaskCompleted` | 태스크 완료 처리 시 | 완료 차단 후 피드백 전송 |

## 6. 내부 아키텍처

### 구성 요소

| 구성 요소 | 역할 |
|---|---|
| **Team lead** | 팀 생성, 팀원 스폰, 작업 조율하는 메인 세션 |
| **Teammates** | 각자 태스크를 처리하는 독립 Claude Code 인스턴스 |
| **Task list** | 팀원이 클레임하고 완료하는 공유 작업 목록 |
| **Mailbox** | 에이전트 간 메시지 라우팅 시스템 |

### 저장 위치 (런타임 동안만 존재)

```
~/.claude/teams/{team-name}/config.json   # 팀 설정 (세션 ID, tmux pane ID 등)
~/.claude/tasks/{team-name}/              # 태스크 리스트
```

> 팀 정리 또는 세션 종료 시 자동 삭제. 수동 편집 금지 (다음 상태 업데이트 시 덮어씀).

### 컨텍스트와 통신 방식

- 팀원은 리드의 대화 히스토리를 상속하지 않음
- 팀원은 CLAUDE.md, MCP 서버, 스킬은 프로젝트 설정에서 자동 로드
- 팀원 메시지는 수신자에게 자동 전달 (리드가 폴링 불필요)
- 팀원 유휴 시 리드에게 자동 알림
- 특정 팀원에게만 메시지 가능 (브로드캐스트 없음, 전체 전달 시 1:1로 여러 번 발송)

### 서브에이전트 정의 재사용

```
security-reviewer 에이전트 타입으로 팀원을 스폰해서 인증 모듈을 감사해줘.
```

서브에이전트 정의의 `tools` 허용 목록과 `model`을 팀원에게 적용 가능.  
단, `skills`와 `mcpServers` 프론트매터는 팀원 모드에서 무시됨.

### 권한

팀원은 리드의 권한 설정을 초기값으로 사용.  
리드가 `--dangerously-skip-permissions`로 실행되면 모든 팀원도 동일하게 적용.

## 7. 사용 사례 예시

### 병렬 코드 리뷰

```
PR #142를 리뷰할 에이전트 팀을 만들어줘.
- 한 팀원: 보안 취약점 집중 분석
- 한 팀원: 성능 영향 분석
- 한 팀원: 테스트 커버리지 검증
각자 리뷰 후 결과를 보고해줘.
```

### 경쟁 가설로 버그 조사

```
사용자가 앱이 한 메시지 후에 연결이 끊긴다고 보고했다.
5명의 팀원을 스폰해서 서로 다른 가설을 조사하게 해줘.
팀원들이 서로의 이론을 반박하도록 과학적 토론 방식으로 대화하게 해.
합의된 결론을 findings 문서에 업데이트해줘.
```

> 핵심: 팀원들이 서로를 적극적으로 반박하게 만들면, 살아남은 이론이 실제 원인일 가능성이 높아진다.

## 8. 베스트 프랙티스

### 팀원에게 충분한 컨텍스트 제공

팀원은 리드의 대화 히스토리를 못 받으므로 스폰 프롬프트에 명시:

```
security-reviewer 팀원을 스폰해줘. 프롬프트: "src/auth/의 인증 모듈을
보안 취약점 관점에서 리뷰해. 토큰 처리, 세션 관리, 입력 검증에 집중.
앱은 httpOnly 쿠키에 저장된 JWT 토큰을 사용함. 발견된 이슈를 심각도 등급과 함께 보고해."
```

### 적절한 팀 규모

- **시작 권장**: 3~5명 (대부분의 워크플로우에서 균형 잡힘)
- 팀원당 5~6개 태스크가 이상적
- 토큰 비용은 팀원 수에 선형 비례
- 협업이 실질적으로 필요할 때만 규모 키울 것

### 태스크 크기 조정

| 너무 작음 | 적당함 | 너무 큼 |
|---|---|---|
| 조율 오버헤드 > 이익 | 명확한 산출물이 있는 독립 단위 (함수, 테스트 파일, 리뷰 등) | 체크인 없이 너무 오래 작업 |

### 팀원 완료 대기

리드가 팀원 대신 직접 구현을 시작하면:
```
팀원들이 태스크를 완료할 때까지 기다려줘.
```

### 파일 충돌 방지

두 팀원이 같은 파일을 편집하면 덮어씌워짐.  
팀원마다 다른 파일 세트를 소유하도록 작업 분배.

### 모니터링과 방향 조정

팀이 오래 방치될수록 낭비되는 작업이 늘어난다.  
진행 상황을 주기적으로 확인하고, 잘못된 방향은 일찍 수정.

## 9. 트러블슈팅

| 문제 | 원인 | 해결 방법 |
|---|---|---|
| 팀원이 나타나지 않음 | tmux 미설치 또는 작업이 팀 생성을 정당화할 만큼 복잡하지 않음 | `which tmux` 확인, `Shift+Down`으로 in-process 팀원 확인 |
| 권한 프롬프트가 너무 많음 | 팀원 권한 요청이 리드에게 버블업 | 팀 생성 전에 공통 작업 권한 미리 승인 |
| 팀원이 오류 후 멈춤 | 오류 복구 실패 | `Shift+Down`으로 출력 확인 후 직접 지시 또는 교체 팀원 스폰 |
| 리드가 작업 완료 전에 종료 | 리드가 팀 완료로 오판 | 계속 진행 요청, 팀원 대기 지시 |
| tmux 세션이 남아 있음 | 팀 정리 불완전 | `tmux ls` 후 `tmux kill-session -t <name>` |

## 10. 알려진 제한사항

| 제한사항 | 내용 |
|---|---|
| **세션 복원 불가** | `/resume`, `/rewind` 시 in-process 팀원 복원 안 됨 |
| **태스크 상태 지연** | 팀원이 태스크 완료 표시를 못할 수 있음 → 수동으로 업데이트 필요 |
| **종료 지연** | 팀원이 현재 요청/도구 호출 완료 후 종료 |
| **팀 하나만** | 리드는 팀 하나만 관리 가능 |
| **중첩 팀 불가** | 팀원은 자신의 팀/팀원을 스폰할 수 없음 |
| **리드 고정** | 팀을 생성한 세션이 영구 리드, 리더십 이전 불가 |
| **스폰 시 권한 고정** | 팀원은 리드의 권한 모드로 시작, 스폰 시 개별 설정 불가 |
| **Split panes 제한** | VS Code 터미널, Windows Terminal, Ghostty에서 미지원 |

## 빠른 참조: 핵심 명령어

```
# 팀 생성
"[작업 설명]을 위해 [N]명의 팀원으로 에이전트 팀을 만들어줘"

# 계획 승인 요구
"[팀원명] 팀원을 스폰하되 변경 전 계획 승인을 요구해"

# 팀원 종료
"[팀원명] 팀원에게 종료 요청해줘"

# 팀원 대기
"팀원들이 태스크를 완료할 때까지 기다려줘"

# 팀 정리
"팀 정리해줘"
```

```bash
# 화면 전환 (in-process 모드)
Shift+Down   # 다음 팀원으로 이동
Ctrl+T       # 태스크 리스트 토글
Escape       # 팀원 현재 작업 중단
```
